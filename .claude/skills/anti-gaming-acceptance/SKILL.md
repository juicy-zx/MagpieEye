# 防蒙混验收体系设计

## Overview

设计验收规则时,必须代入"想用最低成本蒙混过关的执行者"视角,对**每条**信号问:"执行者最便宜的通过路径是什么?"——如果最便宜路径不解决真实问题,这条规则就是取巧通道。宽松的对象只能是**容差阈值**,永远不能是**验证结构**。

## 六条结构性规则

1. **前置依赖硬化**:硬门禁信号的前置/使能信号(标注覆盖率、配对率、可比对集合大小)不足时,结论必须是"**不可判定 → 不通过**"(inconclusive),绝不是警告。前置信号是验证器的眼睛,不是锦上添花的便利品——执行者少打标注,验证器就失明,"零违规"变成"零检出"。
2. **禁止收窄下游掩护前置**:不得通过重定义下游验证范围("只比最终输出,不逐阶段比对")来论证前置信号无关紧要——那是用降低验证分辨率换宽松,恰好放走分辨率本可抓到的 hack(如对基准输入的特判硬编码)。
3. **禁自声明进门禁**:被验属性不得由被验方自己声明后参与 pass 判定(自报清单、自评分)。可作调试 metadata 存证,不得进门禁。
4. **禁同源自证**:验证信号必须独立于被验方——自写测试通过不是证据(选手兼裁判);同一通道不能验证它自己的产物(交叉验证必须两条独立信源)。注意反向激励:自测失败也不应阻断,否则逼出"少写/不写测试"的防御行为——自测信号正确地位是"零权重存证"。
5. **基准防过拟合**:被验方不可知基准全集(保密/轮换/超集抽样),或保持高验证分辨率(逐阶段/逐属性比对)使特判无处藏身。固定且可见的基准集必被过拟合。
6. **防护内建单轮**:取巧防线必须内建于每一轮验收的结构中;"事后人工抽查 10%""版本级信任治理"是运营手段,不是验收防护——把对抗面外包给人工,在自动化闭环里等于不设防。

## 无基准态两分法

有真值基准的属性走 **parity**(逐属性与基准比对);无基准的运行态(超长内容/异常态/加载失败)不跳过验证,走 **invariant**(不比"像不像",只断言"没有当场崩坏":溢出/裁剪/重叠/最小触控)。两路互斥,不得混用,不得因无基准而放行。

## 设计完成后的自检

逐条信号填表:"执行者作弊清单"——每条规则写出至少一种取巧方式与它被哪条结构挡住;挡不住的,硬化它。识别出的取巧方式登记进反模式清单,**只增不删**。

## 反说辞表

| 说辞 | 现实 |
|---|---|
| "即使标注全错也不影响对最终输出的判定" | 你刚刚把验证收窄到失明,才让它"不影响"——中间态 hack 恰好靠被你砍掉的分辨率检出。基线测试原话即此。 |
| "覆盖率只是提升排障效率的辅助" | 它是验证器的眼睛。不足=不可判定=不通过。 |
| "要宽松友好、减少误杀" | 误杀由容差阈值和分级 severity 调节;验证结构上的洞与友好无关,只与可作弊有关。 |
| "有警告+人工抽查兜底" | 抽查是运营,不是验收。单轮内没有结构防护 = 不设防。 |
| "自测失败连续 N 次才阻断" | 给了"干脆不写测试"的出口。自测信号零权重,双向都不进判定。 |
| "渲染/环境失败就当零违规放行" | "跑不了=测不出=通过"是最大的后门;环境失败必须短路为不通过(独立失败类别)。 |

## Red Flags — 出现任一立即重审

- 任何"覆盖率/标注率/配对率低 → 仅警告"
- 被验方自报的任何信号出现在 pass 判定分支里
- 用"只验最终输出"论证中间验证/前置信号可有可无
- 基准集固定、可见、无轮换,且验证只看端到端结果
- "人工抽查""信任分级"出现在防作弊设计里
- 无基准场景被跳过而非走 invariant
