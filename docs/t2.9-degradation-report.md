# T2.9 慢车道性能退化诊断报告(Codex D-08)

**结论先行**:D-08 立项所依据的"M2 后段(T2.6~T2.8)引入 ~1.8s 裸口径退化"与"慢车道 P50=10140ms"
**均非工程本体退化**。裸 Gradle 热路径无退化;10140ms 中约一半是 **odiff-bin 常驻 server 的一枚
未 unref、无句柄可清的 5s 看门狗定时器**在 check 功能完成后空转吊住进程所致。修复该空转尾后,
慢车道热 P50=**5090ms ≤ 6000ms 达标(pass:true)**。npm test 215/31 全绿。

测量环境:JDK 21 (Corretto 21.0.9),Gradle 9.5.1,warm daemon,`GRADLE_USER_HOME=demo-android/.gradle-home`。
每候选/端点 ≥3 轮取中位;官方 P50 用 `scripts/t2.1/measure-hot.mjs`(1 预热 + 10 轮)。

---

## 1. 归因结论

### 1.1 "裸口径 5838→7656 退化"——证伪(测量口径错配 + 冷 daemon 离群)

`5838ms` 的出处经溯源(`.claude/plans/.../orchestration.md`、`latency-m2.json@4fa01f1`):
它记录于 commit **4fa01f1**(M2 review round1,即 T2.6/T2.7/T2.8 全部之后、≈HEAD 的代码),
且由**旧 `measure-hot.mjs` 的 EOF-comment-toggle 协议**产出——该协议两态切换,第二态命中
Gradle build cache,测试任务被 **FROM-CACHE 还原(非真实渲染)**。它**从未在 5c689c4(T2.1 交付)
上测过**;"T2.1 期 5838ms"是误标注。

因此 `5838`(旧协议·缓存命中)与 `7656/10140`(新协议·`--rerun` 真渲)是**同一份代码、
不同测量协议**的对比,非退化。静态佐证:`5c689c4..HEAD` 窗口内 demo-android **仅** 被
`5af4a49`(录入 golden PNG)与 `a8ac8c1`(截图落盘路径 +3/-1 行)触碰;
`libs.versions.toml / gradle.properties / build-logic / settings.gradle.kts / app/build.gradle.kts`
**零改动**,demo-android 编译类路径未增长,fastlane-worker **未**进入 demo-android settings/测试类路径。

`7656ms`(T2.1-fix 记录的"裸单轮")为**半冷 daemon 单轮离群值**;warm 稳定值 ~5.2s(见 §2)。

### 1.2 "P50=10140ms / daemon+CLI ~2.5s 开销"——真因是 odiff 看门狗空转尾(可修复)

慢车道 CLI 热路径全链拆四段时间戳(HEAD,warm,修复前):

- 真实工作(node 启动→report 落盘打印)≈ **5051ms**(与裸/守护渲染同量级);
- **report 打印→进程退出 ≈ 4964ms(每轮恒定,非可变清理)**——纯空转尾,check 早已完成。

对照实验 `CI=1`(odiff-bin `server.js:153` 的 `if (!process.env.CI)` 会跳过看门狗):
report→exit 塌缩到 **3~8ms**,total 5018→**4992ms**。**证实空转尾 100% 来自该定时器**。

根因:`node_modules/odiff-bin/server.js:154` 在 server 初始化时
`setTimeout(()=>{ if(!ready) stop() }, 5000)`——**无变量持有、`stop()/cleanup()` 均不清它、
且未 `unref()`**。server 在 L1 阶段(~4.9s 处)才懒启动,故定时器在 ~9.9s 处才 fire,
把 Node 事件循环吊到 ~10s 才退。旧口径未暴露是因当时未以"进程退出"为计时窗口测全链。

所谓"daemon+CLI ~2.5s 开销"不成立:纯 daemon `gradle.run`(Tooling API,warm ProjectConnection)
= **4901ms**,**比裸 gradlew(5184ms)还快**(省去 gradlew client JVM 启动);node 启动+选路仅 ~110ms,
L1/L2/report ~50~150ms。真实 daemon+CLI 净开销相对裸渲染近乎为零。

### 1.3 其余疑因

- **CLI 退出治理 300ms 宽限**:仅 `SpawnGradleRunner`(**冷路径** spawn)使用;热路径走 UDS,不经过,不计入热 P50。
- **odiff server stop ~5s**:方向对(确是 odiff),但机理是**启动看门狗定时器空转**,非 `stop()` 阻塞;`stop()` 本身仅 `kill()`,不耗时。

## 2. 裸 Gradle bisect(A 段)

`--rerun` 仅强制 `:app:testDebugUnitTest` 重跑(真实 Robolectric 渲染);上游 compile 任务恒 UP-TO-DATE,
config-cache 复用。故裸 P50 = Gradle 启动 + up-to-date 检查 + 真实渲染。

窗口内 demo-android 只有**两个不同构建态**:
态 X(录 golden 前)= {5c689c4, 4120834, 9d79898, …};态 Y(录 golden 后)= {a8ac8c1/5af4a49 起 … 687bae7, HEAD}。
故只需测两端点即完成 bisect;中间"仅动 packages/daemon/fastlane-worker/docs"的候选,其 demo-android
构建输入与相邻端点**逐字节相同**,裸 P50 定义上不变(无需另测)。

此为 `--rerun` 真口径的固有成本(旧 build-cache 曾将其掩盖)。

## 3. CLI 全链分解与修复复测(B 段,HEAD,warm daemon,hot lane)

`UIV_FASTLANE=0 UIV_RERUN=1`,`measure-hot.mjs` 以进程退出为计时窗口(`spawnSync`)。

| 阶段(中位) | 修复前 | 修复后 |
|---|---|---|
| node 启动 → 选路(daemon ping) | 112ms | 96ms |
| 选路 → report 落盘+打印(gradle 渲染 + L1 odiff + L2 + report) | 4939ms | 5016ms |
| **report 打印 → 进程退出(teardown)** | **4964ms** | **2~3ms** |
| 全链合计(wall-clock) | **10022ms** | **5112ms** |
| 对照:纯 daemon `gradle.run`(B2) | 4901ms | 4901ms |
| 对照:`CI=1`(跳过看门狗) | total 4992 / exit 6ms | — |
| **官方 `measure-hot.mjs` P50(10 轮)** | 10140ms (D-07) | **5090ms** (4992~5324, 极差 6.6%) |
| pass @ target 6000 | false | **true** |

### 修复项

`packages/uiv-cli/src/index.ts`:`main()` 结束(成功/异常均含 `finally stopOdiffServer()`)后,
`flushAndExit()` 先 flush stdout(保末行 report/spec 路径契约)再显式 `process.exit(code)`,
消除 odiff 看门狗定时器造成的 ~5s 空转尾。**契合 D-07 退出治理原旨("check 完成后进程短超时内退出"),
不改 L1 常驻 server 架构本身,不降任何功能/覆盖。**

### 回归验证

- `npm test`:**31 files / 215 tests 全绿**(基线 215/31 保持)。
- 快车道 e2e 抽验:默认模式 `check` → `lane=fast`,exit 0,3116ms 干净返回(不再 ~5s 悬挂,报告路径正确)。
- 修复对两车道正交(fast/slow 都经 L1 odiff,都受益于即时退出)。
