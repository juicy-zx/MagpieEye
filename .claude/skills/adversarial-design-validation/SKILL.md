# 对抗性方案设计验证

## Overview

方案的价值上限由其最弱的承重假设决定。**承重断言(若为假则方案塌陷的断言)未经证伪核查,方案不得标"定稿"**——"已标注待验证"不豁免。

## 承重断言识别

扫描方案,凡命中以下任一且"为假则核心路径塌陷"的,列入承重清单:

- 第三方配额/限流数字(尤其来源是"同事说的/文档没查过")
- 性能经验值("通常很快""2 秒内"——单点经验≠分布,长尾未知)
- API/能力**存在性与开通状态**(接口存在 ≠ 你的组织已开通 ≠ 你的 token 有权限 ≠ 覆盖你的场景)
- 平台兼容性/版本行为(升级后是否仍成立)

## 流程(按方案分量裁剪,承重证伪不可裁)

1. **多路调研**——每路一个独立维度;产出格式内建 `riskyClaims`(推测性声明)+ `verdicts`(逐条判决),对抗验证前移。
2. **多视角互斥方案**——按互斥立场(体验优先/严谨优先/务实最小)各出完整架构,不是一版改三遍。
3. **评委评审**——打分必须拉开差距,禁和稀泥(LLM 评委默认收敛 7-8 分);选出胜者的同时强制提取落选方案可嫁接的亮点。
4. **承重断言证伪**——每条承重断言独立派一个核查 agent,指令模板:

   > 你是对抗性事实核查员,任务是**尽力证伪**下面这条技术断言(不是核实它)。只采信官方文档、GitHub release 与 issue 等一手来源,不要只看博客转述;不确定时倾向 uncertain 而非 confirmed。

5. **成稿规则**——所有**承重**断言 verdict ∈ {confirmed, refuted→已改写受影响段落} 才能标"定稿"。uncertain 断言只有两条出路:①阻塞定稿(文档标"草稿(定稿阻塞项:…)"+ 每项验证方法);②**显式移出承重路径**——改写架构使主路径不再依赖它(降级为可选优化/兜底),并在核查表里记录移出方式;口头宣称"这条不承重"而主路径仍依赖它 = 违规。另:**只能现场验证的运行时状态**(组织开关是否已启用、账号权限是否已授予)不属于"查文档可证伪"类,必须转化为上线门槛项(Day 0 checklist)+ 架构防御(如把"未启用"与"检查通过"建模为不同状态,防静默假阴性),不得只标一句"待确认"。单点实测与推断在脚注区分置信度。

## 反说辞表

| 说辞 | 现实 |
|---|---|
| "已标注'假设-待验证',不影响今天定稿" | 定稿的语义就是"可据此开工";标注不会阻止下游按定稿执行。基线测试原话即此,且其新架构正建立在自己标了"待确认风险项"的假设上。 |
| "时间紧,今天必须定稿" | 错误前提会固化进所有上层组件。实证:某方案"桌面通道免配额"被证伪→基准层整体重构;"暖 daemon 秒级"被证伪→内循环目标全改。一次证伪(半天)≪ 返工(数周)。 |
| "我做了保守估算/敏感性分析" | 敏感性分析建立在"假设为真"的分布上;证伪问的是"它压根不可用/未开通怎么办"。 |
| "这是官方 API,肯定能用" | 存在 ≠ 已开通 ≠ 有权限 ≠ 覆盖你的场景。 |
| "多视角/评委太重了" | 可按分量裁剪 2/3 阶段;但承重断言证伪(第 4 步)与成稿规则(第 5 步)不可裁。 |
| "这条 uncertain 不承重,可以定稿" | 必须拿出移出承重路径的架构证据(主路径已改为不依赖它);拿不出 = 它仍承重,阻塞定稿。 |

## Red Flags — 出现任一立即停下

- 方案文本含"待验证但不阻塞""不影响定稿"
- 数字来源是"同事说的""基于经验""通常"
- 性能预算表引用从未实测的数字
- 新主路径依赖的能力,自己在风险表里标了"待确认"
- 单方案直出且无承重断言清单
